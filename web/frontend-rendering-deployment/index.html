フロントエンドのレンダリングとデプロイ形式を比較する

CSR、SSG、SSR、ISR、Edge SSR、Islands Architecture、React Server Components、PWAについて、どこでHTMLを作り、どこへデプロイし、どんなトレードオフがあるのかを整理する。

結論

更新頻度が低く、認証や個別表示が少ないページはSSGが第一候補です。ユーザーごとに内容が変わる、検索流入も重視する画面はSSRまたはEdge SSRを検討します。アプリ的な操作が中心でSEO要求が薄い管理画面はCSRで十分なことが多いです。記事や商品詳細のように「多くは静的だが、ときどき更新される」ページはISRや再検証付きのキャッシュが合います。

近年の実務では、1つに決め打ちするより、ページ単位・コンポーネント単位で混ぜる設計が主流です。静的HTMLを土台にし、必要な部分だけサーバー実行やクライアント実行を足すほど、性能、運用、開発体験のバランスを取りやすくなります。

用語の前提

レンダリング形式とは「HTMLやUIをいつ、どこで作るか」です。デプロイ形式とは「生成物をどこに置き、リクエスト時に何を実行するか」です。たとえばSSGはビルド時にHTMLを作り、静的ホスティングやCDNへ置けます。SSRはリクエスト時にサーバーやエッジでHTMLを作るため、実行環境とキャッシュ設計が必要です。

全体比較

形式 HTMLを作る場所 主なデプロイ先 向いている用途
CSR ブラウザ 静的ホスティング、CDN 管理画面、SaaS、ログイン後アプリ
SSG ビルド時 静的ホスティング、CDN、GitHub Pages ブログ、ドキュメント、LP、企業サイト
SSR リクエスト時のサーバー Node.js、サーバーレス、コンテナ 検索流入が必要な動的ページ、認証つきページ
ISR / 再検証 ビルド時 + 期限後に再生成 対応フレームワークのホスティング 商品詳細、ニュース、CMS記事

Islands 静的HTML + 必要箇所だけブラウザまたはサーバー 静的ホスティング、サーバーレス、Node.js コンテンツ中心だが一部だけインタラクティブなサイト
RSC サーバーコンポーネントはサーバー、必要箇所はクライアント 対応フレームワークのサーバー環境 Reactでデータ取得とUI分割を細かく設計するアプリ
PWA 形式ではなく配布・実行拡張 通常のWebホスティング + manifest / service worker オフライン対応、ホーム画面追加、アプリ風体験

CSR

CSR、Client Side Renderingは、最初にほぼ空のHTMLとJavaScriptを配信し、ブラウザ上で画面を組み立てる方式です。Viteなどでビルドした成果物を静的ファイルとして配信する構成が典型です。

メリット

  • 静的ファイル配信だけで動くため、サーバー運用が軽い。
  • ログイン後の複雑な状態管理や画面遷移を作りやすい。
  • CDNに載せやすく、デプロイ構成が単純になりやすい。

デメリット

  • 初回表示までにJavaScriptの取得、解析、実行が必要になる。
  • 検索エンジンやSNSプレビュー向けのHTMLが弱くなりやすい。
  • 低スペック端末ではJavaScript実行の負荷が体感性能に出やすい。

管理画面や社内ツールのように、検索流入より操作性が重要な画面では今でも現実的な選択です。一方、公開ページ全体をCSRだけにすると、SEO、初期表示、メタデータ制御で不利になる場面があります。

SSG

SSG、Static Site Generationは、ビルド時に各ページのHTMLを作っておく方式です。生成後はHTML、CSS、JavaScript、画像を静的ファイルとして配信できます。

メリット

  • リクエスト時にアプリサーバーを動かさないため、速く、落ちにくく、安く運用しやすい。
  • HTMLが最初から存在するため、SEOやSNSプレビューと相性がよい。
  • GitHub Pagesのような静的ホスティングにそのまま載せやすい。

デメリット

  • 更新のたびにビルドとデプロイが必要になる。
  • ページ数が多いとビルド時間が伸びる。
  • ユーザーごとの認証情報、Cookie、リアルタイム在庫などは静的HTMLだけでは扱いにくい。

このノート集のような静的HTMLサイトはSSGまたは手書き静的サイトに近い構成です。内容が公開情報で、更新頻度が高すぎないなら、最も堅い選択になりやすいです。

SSR

SSR、Server Side Renderingは、リクエストごとにサーバーでHTMLを生成し、ブラウザへ返す方式です。ReactではサーバーでHTMLをストリームとして生成し、クライアント側でhydrateしてインタラクティブにするAPIがあります。

メリット

  • 初回レスポンスにHTMLを含められるため、検索流入や共有リンクに強い。
  • ユーザー、Cookie、リクエストヘッダー、地域などに応じた表示を作りやすい。
  • データ更新が頻繁でも、ビルドを待たずに最新情報を返せる。

デメリット

  • サーバー実行環境、監視、スケール、キャッシュ設計が必要になる。
  • 遅いAPIやDBに引きずられると、HTMLの返却自体が遅くなる。
  • hydration用JavaScriptが多いと、見えているのに操作できない時間が残る。

SSRは「SEOのために万能」というより、リクエスト時の情報をHTMLに反映したい場合の選択肢です。キャッシュできる部分と毎回計算する部分を分けないと、サーバー負荷と遅延が増えます。

ISR / 再検証

ISR、Incremental Static Regenerationは、静的生成したページを一定条件で再生成する考え方です。フレームワークによって名前やAPIは違いますが、「普段は静的に速く返し、期限切れやイベントで再生成する」構成と考えると理解しやすいです。

メリット

  • SSGの速さとSSRの更新追従性をある程度両立できる。
  • 大量ページを毎回フルビルドしなくてよい。
  • CMSや商品データの更新に合わせて段階的に反映できる。

デメリット

  • 対応ホスティングやフレームワークのキャッシュ機構に依存しやすい。
  • 古い内容が一定時間表示される可能性がある。
  • 「いつ再生成されたか」「失敗時に何を見せるか」の運用設計が必要になる。

在庫、価格、法務文言のように古い表示が許されない情報には注意が必要です。多少の遅延更新を許容できる記事、商品説明、カテゴリ一覧などに向きます。

Edge SSR

Edge SSRは、ユーザーに近いエッジロケーションでリクエスト時レンダリングを行う方式です。Node.jsサーバーより低遅延にしやすい一方、実行時間、使用できるAPI、パッケージ互換性に制約が出ることがあります。

メリット

  • ユーザーに近い場所でHTMLを作れるため、地域差のあるサービスで遅延を抑えやすい。
  • A/Bテスト、リダイレクト、地域別コンテンツ、軽い認証分岐と相性がよい。
  • CDNキャッシュと組み合わせると、動的処理と高速配信を両立しやすい。

デメリット

  • Node.js前提のライブラリがそのまま動かない場合がある。
  • 重いDB接続や長時間処理には向きにくい。
  • ローカル開発、ログ、監視、障害調査が通常のサーバーより複雑になることがある。

「全ページをEdgeでSSRする」より、低遅延が効く入口ページや軽いパーソナライズに絞る方が扱いやすいです。

Islands Architecture

Islands Architectureは、ページの大半を静的HTMLとして出し、検索ボックス、カルーセル、カートなど必要な部品だけJavaScriptでhydrateする考え方です。Astroはこの考え方を中心にしたフレームワークです。

メリット

  • ページ全体をSPA化しないため、クライアントJavaScriptを減らしやすい。
  • 重要な静的本文を先に表示し、操作部品だけ後から読み込める。
  • コンテンツサイトに一部だけリッチなUIを足す構成と相性がよい。

デメリット

  • 島同士の状態共有が増えると設計が難しくなる。
  • アプリ全体が高度に連動する画面では、通常のSPAやSSRアプリの方が自然な場合がある。
  • チームが従来のSPA前提だと、どこをhydrateするかの設計に慣れが必要になる。

ブログ、メディア、ECの商品ページ、ドキュメントのように「読む部分が主役で、一部だけ動く」サイトでは強力です。

React Server Components

React Server Components、RSCは、Reactコンポーネントをサーバー側で実行するものとクライアント側で実行するものに分ける仕組みです。Next.js App Routerでは、レイアウトやページは標準でServer Componentsとして扱われ、ブラウザAPIやイベント処理が必要な部分にClient Componentsを使います。

メリット

  • DBや内部APIに近い場所でデータ取得し、不要なクライアントJavaScriptを減らしやすい。
  • ページ全体ではなくコンポーネント単位でサーバーとクライアントの責務を分けられる。
  • ストリーミングやキャッシュと組み合わせると、重い部分を待たせず段階的に表示できる。

デメリット

  • サーバーで動くコードとブラウザで動くコードの境界を意識する必要がある。
  • 対応フレームワークの設計思想に依存しやすい。
  • 従来のReact SPAだけを知っている場合、データ取得、キャッシュ、コンポーネント分割の学習コストがある。

Reactで大きなアプリを作る場合、RSCはSSRの単なる別名ではありません。ページ全体のレンダリング方式ではなく、コンポーネントの実行場所を分ける設計要素として見るのが適切です。

PWA

PWA、Progressive Web Appは、レンダリング方式ではなく、Webアプリをインストール可能にしたり、オフライン体験を提供したりする配布・実行面の拡張です。manifestやservice workerを使い、通常のWebサイトをアプリ風に扱えるようにします。

メリット

  • アプリストアに限らずWebから配布でき、ホーム画面やDockから起動できる場合がある。
  • service workerでキャッシュやオフライン表示を設計できる。
  • 同じWebコードベースをアプリ風の体験に近づけられる。

デメリット

  • ブラウザやOSによって対応機能とインストール体験が異なる。
  • キャッシュ戦略を間違えると、古い画面や古いJavaScriptを配り続ける事故が起きる。
  • ネイティブアプリと同じ権限やUXを常に実現できるわけではない。

PWAはCSR、SSG、SSRのどれとも組み合わせられます。オフライン要件、再訪頻度、通知、ホーム画面起動が重要な場合に検討します。

選び方

  1. 公開コンテンツ中心ならSSG。 ブログ、ドキュメント、LP、会社サイトはまず静的化を考える。
  2. ログイン後の操作中心ならCSR。 SEOが不要でAPI経由の状態変化が主役なら、静的配信のSPAで十分なことが多い。
  3. リクエストごとにHTMLを変えるならSSR。 Cookie、認証、地域、検索条件、在庫などを初回HTMLへ反映したい場合に使う。
  4. 静的だが更新も必要ならISR。 古い表示を短時間許容できるかを先に決める。
  5. 本文中心で一部だけ動くならIslands。 JavaScriptをページ全体へ広げない。
  6. Reactで大規模化するならRSCも検討。 サーバーでよい処理をクライアントへ送らない設計にする。
  7. アプリ風配布が必要ならPWAを足す。 レンダリング方式とは別軸で検討する。

判断表

条件 有力候補 避けたい選択
SEOが重要で内容があまり変わらない SSG、Islands CSRのみ
ログイン後の業務アプリ CSR、SSR + CSR 全画面SSGのみ
商品ページが大量にあり、ときどき更新される ISR、SSG + on-demand再検証 毎回フルビルドだけに依存
地域別表示や軽いパーソナライズが必要 Edge SSR、SSR 静的HTMLだけで無理に分岐
ネットワーク不安定な環境でも使わせたい PWA + 適切なキャッシュ キャッシュ戦略なしのSPA
